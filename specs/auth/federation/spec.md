# Federation (DFN-AAI / eduGAIN)

## Purpose

SAML federation support for German research organizations via the DFN-AAI identity
federation and eduGAIN inter-federation. Users from federated institutions (DFN
members, international partners) authenticate with their home institution
credentials via Shibboleth IdP.

## Non-Goals

- Keycloak as SAML IdP for federation (use Shibboleth IdP)
- Non-DFN federation configurations (custom IdPs configured separately)

## Requirements

### Requirement: DFN-AAI Service Provider registration

The platform SHALL operate as a SAML SP within the DFN-AAI federation, generating
standard SAML metadata for federation registration.

The target SP entity ID SHALL be `https://idp.opendesk.hrz.uni-marburg.de`.

#### Scenario: Federated user authenticates via discovery service
- GIVEN a user from a federated university (e.g., LMU Munich)
- WHEN the user navigates to the platform's federation login endpoint
- THEN the user is presented with the DFN-AAI Discovery Service
- AND the user selects their home institution (LMU Munich Shibboleth IdP)
- THEN the user is redirected to the homeinstitution's login page
- AND upon authentication, the user is redirected back with a SAML assertion
- AND SAML attributes are released to the platform (via entity ID mapping)
- AND the user gains access without a local account

#### Scenario: eduGAIN user authenticates via inter-federation
- GIVEN a user from an international institution (e.g., ETH Zurich) participating in eduGAIN
- WHEN the user selects their home institution in the DFN discovery service
- THEN the request flows via the DFN-AAI → eduGAIN bridge
- AND the ETH Zurich Shibboleth IdP authenticates the user
- AND the platform receives a valid SAML assertion with eduGAIN attributes

### Requirement: DFN-AAI metadata registration

The platform SHALL register as a DFN-AAI SP with the following entity ID and
metadata location:

| Property | Value |
|----------|-------|
| Entity ID | `https://idp.opendesk.hrz.uni-marburg.de` (Shibboleth IdP) |
| Metadata URL | `https://idp.opendesk.hrz.uni-marburg.edu/metadata` (generated by Shibboleth) |
| Certificate | Institutional TLS certificate (HRZ Marburg) |
| SLS | `https://idp.opendesk.hrz.uni-marburg.de/idp/shibboleth` |

#### Scenario: Metadata fetch by federation
- GIVEN the DFN-AAI federation metadata aggregator
- WHEN the aggregator fetches `https://idp.opendesk.hrz.uni-marburg.de/metadata`
- THEN valid SAML 2.0 metadata is returned
- AND the `entityID` matches `https://idp.opendesk.hrz.uni-marburg.de`
- AND the `SigningCertificate` is the institutional TLS cert

### Requirement: Shibboleth IdP deployment

The platform SHALL deploy a Shibboleth IdP as the federation gateway, NOT Keycloak.

Keycloak acts only as a user store (LDAP-backed) while Shibboleth handles
inter-federation SAML flows.

#### Scenario: Shibboleth IdP proxying to Keycloak
- GIVEN a Shibboleth IdP deployed in the `opendesk` namespace
- WHEN a federated authentication request arrives
- THEN Shibboleth presents the DFN-AAI discovery service
- AND upon selecting a home IdP, Shibboleth initiates SAML SP-SSO
- AND after receiving the assertion, Shibboleth creates a local Keycloak session
- AND the user is redirected to the application with a Keycloak SSO token

### Requirement: eduGAIN attribute mapping

The platform SHALL support the following standard eduGAIN attributes for
federated identity mapping to Keycloak:

| eduGAIN Attribute | Keycloak User Attribute | Use Case |
|-------------------|------------------------|----------|
| `mail` | `email` | Primary email |
| `displayName` | `firstName` + `lastName` | Display name split |
| `eduPersonPrincipalName` | `username` | Stable unique identifier |
| `eduPersonAffiliation` | `groups` (mapped) | Group membership derivation |
| `eduPersonEntitlement` | `roles` | Role derivation (optional) |

#### Scenario: Attribute mapping on first login
- GIVEN a federated user authenticating for the first time
- WHEN the SAML assertion contains `eduPersonPrincipalName: user@uni-x.de`
- AND `mail: alice@uni-x.de`
- AND `displayName: Alice Schmidt`
- AND `eduPersonAffiliation: member`
- THEN Keycloak creates a new user with `username: user@uni-x.de` (EPPN)
- AND `email: alice@uni-x.de`
- AND `firstName: Alice`, `lastName: Schmidt`
- AND the user is added to the `managed-by-attribute-federated-users` group

#### Scenario: Unrecognized attributes ignored
- GIVEN a federated authentication event with unsupported attribute `customAttr`
- WHEN the attribute is processed
- THEN it SHALL be ignored
- AND the authentication SHALL NOT fail

### Requirement: Attribute release policy

The platform SHALL request the following attributes from federation IdPs:

- Required: `mail`, `eduPersonPrincipalName`
- Optional: `displayName`, `eduPersonAffiliation`, `eduPersonEntitlement`

#### Scenario: Attribute request in metadata
- GIVEN the DFN-AAI SP metadata
- WHEN an IdP processes the metadata
- THEN the `AttributeConsumingService` lists `mail` as required
- AND `displayName`, `eduPersonAffiliation` are marked as optional

### Requirement: External Shibboleth IdP support

The platform SHALL support connecting to an external Shibboleth IdP as an
additional identity provider in Keycloak, for universities that operate their
own Shibboleth IdP alongside the platform.

#### Scenario: External IdP login via IdP broker
- GIVEN Keycloak configured with an external Shibboleth IdP broker (e.g., ETH Zurich)
- WHEN a user selects the external IdP on the Keycloak login page
- THEN the user is redirected to the ETH Zurich Shibboleth IdP
- AND upon authentication, the external IdP sends a SAML assertion to Keycloak
- AND Keycloak maps the assertion to a local user session via `federated-identity` endpoint
- AND the user gains access to platform services

### Requirement: Persistent NameID format

The platform SHALL use SAML `persistent` NameID format for stable user identity
across sessions.

#### Scenario: Persistent NameID user tracking
- GIVEN a federated user with `eduPersonPrincipalName: user@uni-x.de`
- WHEN the first SAML assertion is received
- THEN the `NameID` format is `persistent`
- AND a hash of the EPPN is stored as the persistent identifier
- AND subsequent logins use the same `NameID` (same user)

## Service Registry

Federation affects:

| Service | Role | Integration Type |
|---------|------|------------------|
| Shibboleth IdP | Federation gateway | SAML SP (to DFN-AAI) |
| Keycloak | User store + IdP broker | SAML SP (to Shibboleth), user federation |
| SAML SP applications | Federation-aware apps | SAML SP (via Shibboleth IdP) |

SAML SP applications (ILIAS, Moodle, BigBlueButton, BookStack, Zammad) inherit
federation login via Shibboleth IdP → Keycloak flow; no per-app configuration
required.

## DFN-AAI Registration Checklist

### Pre-Deployment
- [ ] Shibboleth IdP deployed with institutional TLS certificate
- [ ] Entity ID registered in DFN-AAI portal
- [ ] Metadata URL accessible (`https://idp.opendesk.hrz.uni-marburg.de/metadata`)
- [ ] Attribute release policy configured
- [ ] Test login with DFN-AAI test IdP

### Post-Deployment
- [ ] Discovery service loads all federation IdPs
- [ ] EduGAIN inter-federation enabled (optional)
- [ ] Attribute mapping verified (mail, EPPN, displayName)
- [ ] Persistent NameID login from another DFN institution succeeds
- [ ] External Shibboleth IdP tested via Keycloak IdP broker

## Known Issues

### eduGAIN delay

Fresh eduGAIN attribute mappings may take 24-48 hours to propagate to the
inter-federation bridge. Test with DFN member institutions first.

### Corporate IdPs

Some corporate IdPs (e.g., from German companies) do NOT release all required
attributes. Attribute request failures are handled gracefully but may result in
limited profiles.

## Depends On

- Shibboleth IdP (federation gateway, NOT Keycloak)
- Keycloak (user store, IdP broker)
- DFN-AAI registration (external)
- Institutional TLS certificate (HRZ)
